The tabs and the recent items issues


My take on the tabs and the recent items issues: 


Problems of tabs

As I see it, without much knowledge yet, Entropy5 is not at all simple, speaking of one page. If the user had many pages at once, hidden inside tabs, then complexity might become hard to manage by normal people. 

I agree with Javier Scayola's points, except with the relative importance of the memory issue. I foresee darker scenarios, like a user that unknowingly has an item opened in two tabs, depicting different information in each one because she updated only one and saved it while the other is still in its initial state. The user would say Oops! and repeat the update, possibly damaging the data. To avoid this we should have the server synchronizing different versions of the same record that are rendered in different places, like two tabs of a single user in different locations. 
For example, what to do when a user is browsing a records listing and another user does changes to its data? 

Anyway, IMO the problem is that it is difficult for a user not to lose control when she has lots of data that is not actually visible (or scrollable in sight). GUI applications are good in that they display the information for the user to work on it, instead of making the user to store the context in human memory (as the command line does).
 
What to do?
Before delving into the development of a feature we need to identify the related use cases or user journeys, else we risk implementing a developer's pipe dream and throw ourselves into an unneeded budget overrun. IMO the tabs thing could be a huge one, and if there is a specific usage case then we should revise it to find a better solution. 

The solution is simply to analyse the tasks where this feature appears to be needed, and redesign them so that the UI provides the user with all the information she needs for each step along the journey, all the information, and (almost) only the information needed for the task. 

Recent items
The same as with the tabs, the recent items thing competes with the browser's history and back button, that are already implemented and for free. 

As a web application, Entropy5 must manage its history. This means that each and every time the user moves forward, the application must know if it's appropriate to store the related URL in the browser's history or not. 
Later on, if a user wants to trace back her steps, she can do it by repeatedly clicking the back button, or by clicking and holding to display the recent history. 

One idea could be to add to the product docs a page with a name like Browser tips to get the most of Entropy5 where we can teach the users about tabs, the back / history button, and anything else that could be useful for them.

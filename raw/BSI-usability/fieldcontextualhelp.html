Contextual help example

This is a working prototype to demonstrate a proposal for the form-level and field-level help artifacts.

As of the UI, the help artifacts should be almost invisible, to avoid presenting the user a cluttered screen. The help buttons, in this prototype are small, light gray, and fully displaced to the right.
Anyway, the users will know that they are there, and will be able to find them when the need arises.
When a field gets the focus, the corresponding help icon changes a bit. This small difference tells the users which is the one to click.

Sample form with help

Form level information or help

This place is for form-level information. The opening icon is set near the form's headings.

What follows are some thoughts about help usability.

One idea is not to use too much the word help because users are used to avoid it due to the generalised bad quality.

This can't be the manual. This place is to allow the user to quicly check the purpose and possible effects of this form, by reading few sentences of semantic information.
The manual is somewhere else, and I think it could be valuable to include a link to the approrpiate manual page, for users who are willing to learn.

Most people will be in a hurry. They will be pursuing a goal that is not to learn a lot about the form.

The help sections must have a distinctive style (TBD) for the user to identify them immediately. In this prototype it is different from the form content in that the text is gray and there is a big question mark icon at the left.

Name*

Information on a field doesn't have to be plain text, it can contain:

  • images,
  • graphics,
  • rich text,
  • any other illustrative material.

Actually, it can be an HTML snippet containing any element, including links that should open in a different window.

Last name*

This field, the family name, could have been rendered together with the given name, in the same line, because they are closely related fields.

The first-name / second-name naming structure is something that should'n be taken for granted: in many countries this is not so.

To get a glimpse of the posible alternatives please read an interesting article by the W3C about personal names around the world.

Address*


This is a structured field. It is conceptually one datum although it spans several input fields.

For the purpose of the help it has to be considered a unit, so it's labeled Address and its sub-fields are grouped.

It has to be handled as a unit to avoid annoying and useless instructions like "Enter the address street and number", "Enter the address ZIP code", "Enter the address ...", etc.

In this prototype the sub-fields look somehow scrambled because there is no special alignment imposed. In a real form they should look tidier. One purpose is to raise awareness about the issues that rear when setting several fields of different sizes in a form.

Functionality

The help information is not displayed when the user first sees the form. It is assumed that the users will be recurrent and will need to read the help information about once in their lifetimes, if ever. Thus, the help-related artifacts must not be conspicuous until the users need them.
Also, the help texts should not be set in the form until after the user requests them, but pulled from the server usint XHR actions. This is so to aknowledge the few times the texts are going to be read.

Implementation

The help buttons can be implemented as a custom control that displays the icon and knows an internal field name. When a click happens the control requests the appropriate text in the appropriate language from the server, sets it below the associated field, and displays it using an animtion. The other fields slide downwards slowly to inform the user about what is going on, avoiding the annoying surprise produced by a fast change.

The help texts can be stored in a database, as HTML snippets, related to the corresponding field name stored in the instance of the control.

Content

All instances of the same field set in different forms tipically share the same help text.

Not all fields deserve a contextual help artifact.

Avoid patronizing the users, at the expense of their own time. The after effect of too detailed a help is that users stop using it, forever. Only significant information has to be there.

Optional record fields

The optional user-defined fields must have the possibility of having help too.
In the field definition form there must be an optional field allowing the user to enter the field's help text.
For the user to leverage the HTML capabilities, this field must be a rich text editor capable of producing HTML.
